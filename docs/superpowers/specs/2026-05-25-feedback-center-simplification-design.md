# 反馈中心操作精简设计

日期：2026-05-25  
项目：MerchantWeb 商户外部端  
范围：仅 MerchantWeb 前端反馈中心页面；不改中转平台管理端、不改 Zulin-Monitor 自动开通逻辑。

## 背景

当前反馈中心在手机端信息密度偏高：

- 顶部有 `商品申请 / 问题 / 价格反馈` 主切换。
- `问题 / 价格反馈` 内部又有 `问题反馈 / 价格反馈` 二级切换。
- 问题反馈下还有 5 个问题类型按钮。
- 商品申请页也有较多说明文字，部分说明更像平台内部流程，不适合展示给商户。

结果是商户进入页面后会看到很多按钮和说明，需要先理解分类体系，再决定填写哪个表单。这个体验不够像“提交诉求”，更像“配置后台”。

## 目标

1. 减少手机端按钮数量，让商户更快判断“我要申请商品”还是“我要反馈问题”。
2. 保留现有业务能力：商品申请、普通问题反馈、价格建议反馈。
3. 不向商户暴露平台内部概念，例如关键词映射、默认权重、派单平台配置、商户映射。
4. 表单字段按需出现，避免商户一进来看到过多字段。
5. 保持现有 UI 风格：浅绿色选中态、圆角卡片、紧凑分段切换、移动端友好。

## 非目标

- 不实现中转平台管理端审批卡片。
- 不实现 Zulin-Monitor 中的商品映射、权重、自动开通逻辑。
- 不改变现有商品申请 API 契约。
- 不新增截图上传、附件上传、客服会话等功能。
- 不重构整个 `App.vue` 文件结构；只做反馈中心范围内的局部改造。

## 推荐方案

采用“一级入口不变，内部扁平化”的方案。

### 一级入口

保留顶部两个主入口：

- `商品申请`
- `问题 / 价格反馈`

这两个入口代表商户的真实意图：一个是申请商品，一个是反馈问题。这个层级清晰，继续保留。

### 商品申请页

商品申请页保留两种申请路径，但视觉上要更轻：

- `申请已有商品`
- `申请新增商品`

说明文字压缩为商户能理解的业务语言：

- `已有商品：商品库中存在的商品名称`
- `新增商品：商品库中没有的商品，提交后由平台处理`

申请已有商品时：

- 显示可申请商品列表。
- 如果没有可申请商品，只显示：`暂无可申请商品，请切换到“申请新增商品”。`
- 申请说明 placeholder 使用：`例如：我有该设备，可接单。`

申请新增商品时：

- 显示商品名称、品牌/型号补充、申请说明、联系方式。
- 不出现“关键词库”“派单平台”“商户映射”等内部词。
- 申请说明 placeholder 使用：`例如：我有这台机器，希望平台添加。`

### 问题 / 价格反馈页

删除当前内部的 `问题反馈 / 价格反馈` 二级切换。改为一个统一表单：

1. 顶部标题：`问题 / 价格反馈`
2. 一组问题类型按钮：
   - `页面问题`
   - `派单问题`
   - `接单问题`
   - `价格建议`
3. 商户选择不同问题类型后，表单动态变化。

普通问题类型（页面、派单、接单）只显示：

- 问题描述
- 联系方式（选填）
- 提交按钮

价格建议类型显示：

- 商品
- 当前问题：`推荐价格不合理 / 商品型号不准 / 其他`
- 建议价格：仅当当前问题为 `推荐价格不合理` 时显示
- 原因说明
- 联系方式（选填）
- 提交按钮

这样会从“先选问题反馈或价格反馈，再选具体类型”变成“直接选具体问题类型”。商户少理解一层。

## 交互设计

### 默认状态

- 进入反馈中心默认停留在 `商品申请`。
- 商品申请默认停留在 `申请已有商品`。
- 问题 / 价格反馈默认选中 `页面问题`。

### 切换行为

- 顶部 `商品申请 / 问题 / 价格反馈` 继续使用现有绿色滑块动效。
- 商品申请内的 `申请已有商品 / 申请新增商品` 保持现有 choice chip 风格，但文案下方有一行小字解释。
- 问题类型按钮使用现有 choice chip 风格。
- 选择 `价格建议` 时，价格相关字段淡入显示；选择其他类型时隐藏价格字段。

### 移动端布局

- 保持单列卡片。
- 表单按钮高度不增加，优先减少文案。
- `我的提交记录` 保留在表单下方。
- 如果屏幕高度不足，记录区域允许自然下滑，不强行挤在首屏。

## 文案规范

### 允许展示给商户的词

- 商品库
- 平台处理
- 审批
- 开通商品
- 申请新增商品
- 价格建议
- 问题反馈

### 禁止展示给商户的词

- 关键词映射
- 默认权重
- 权重 1
- 派单平台新增机器
- 商户映射
- Zulin-Monitor
- 中转平台内部处理

### 关键文案

商品申请标题说明：

```text
申请开通已有商品，或提交新增商品申请。
```

已有商品说明：

```text
已有商品：商品库中存在的商品名称
```

新增商品说明：

```text
新增商品：商品库中没有的商品，提交后由平台处理
```

已有商品空态：

```text
暂无可申请商品，请切换到“申请新增商品”。
```

已有商品申请说明 placeholder：

```text
例如：我有该设备，可接单。
```

新增商品申请说明 placeholder：

```text
例如：我有这台机器，希望平台添加。
```

问题 / 价格反馈说明：

```text
选择问题类型后填写说明，价格建议会显示商品和价格相关字段。
```

## 数据流

### 商品申请

已有商品申请沿用现有接口：

- `GET /api/merchant/product-applications/options`
- `POST /api/merchant/product-applications`

新增商品申请沿用同一个提交接口：

- `POST /api/merchant/product-applications`

前端只负责提交申请和展示记录。审批通过后的开通、映射、权重等动作由后端平台完成，不在商户端显示。

### 问题 / 价格反馈

继续使用现有反馈接口：

- `GET /api/merchant/feedback`
- `POST /api/merchant/feedback`

表单映射规则：

- `页面问题 / 派单问题 / 接单问题` 提交为 `type: 'issue'`
- `价格建议` 提交为 `type: 'price_suggestion'`

### 我的提交记录

继续统一显示两类来源：

- 商品申请记录
- 问题 / 价格反馈记录

记录区域仍叫 `我的提交记录`。每条记录用类型标签区分来源。

## 错误处理

- 商品申请选项加载失败：显示已有友好的 `反馈服务暂不可用，请稍后重试` 或对应业务失败文案。
- 商品申请选项为空：显示 `暂无可申请商品，请切换到“申请新增商品”。`
- 普通问题反馈没有填写描述：提示 `请先填写问题描述`。
- 价格建议未选择商品：提示 `请先选择商品`。
- 推荐价格不合理但未填写有效整数价格：提示 `请填写有效的整数建议价格`。
- 非价格字段不显示时，不校验隐藏字段。

## 测试策略

更新现有契约测试，覆盖以下行为：

1. 反馈中心仍有 `商品申请 / 问题 / 价格反馈` 顶部入口。
2. 商品申请页显示 `已有商品：商品库中存在的商品名称`。
3. 已有商品说明 placeholder 是 `例如：我有该设备，可接单。`。
4. `问题反馈 / 价格反馈` 二级切换被删除。
5. 问题类型按钮包含 `页面问题 / 派单问题 / 接单问题 / 价格建议`。
6. 选择价格建议时仍保留价格相关字段契约。
7. 商户端源码中不出现禁止展示词：关键词映射、默认权重、权重 1、派单平台新增机器、商户映射、Zulin-Monitor。

验证命令：

```powershell
npm.cmd test
npm.cmd run build
```

## 实施顺序

1. 更新 `tests/app-contract.test.ts`，先让当前按钮层级和文案测试失败。
2. 修改 `src/App.vue` 的反馈中心模板：
   - 删除问题 / 价格反馈内部二级切换。
   - 将价格建议合并进问题类型按钮。
   - 精简商品申请说明。
3. 复用现有提交函数，新增轻量分流函数或计算属性，避免重写 API。
4. 如有必要，微调 `src/styles.css`，让问题类型按钮在手机端更紧凑。
5. 跑 `npm.cmd test`。
6. 跑 `npm.cmd run build`。

## 验收标准

- 手机端反馈页首屏按钮数量明显减少。
- `问题 / 价格反馈` 中不存在 `问题反馈 / 价格反馈` 二级按钮。
- 价格建议仍能提交价格相关信息。
- 普通问题反馈只显示必要字段。
- 商品申请页不暴露平台内部实现词。
- 所有契约测试和构建通过。

## 风险与取舍

- 删除 `登录账号问题` 独立按钮后，账号问题需要商户选择 `页面问题` 或在描述中说明。这个取舍换来更少按钮和更轻页面。
- `价格建议` 变成问题类型之一后，代码需要确保隐藏字段不参与校验。
- 当前反馈中心仍在 `App.vue` 中实现，本次不做组件拆分，避免扩大改动范围。

## 自审结果

- 无占位符。
- 设计范围只包含 MerchantWeb 前端。
- UI 文案和数据流没有互相冲突。
- 内部实现词只允许出现在本设计文档，不允许出现在商户端页面源码。
